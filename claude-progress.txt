### Implementation Details

**Caching Strategy:**
1. Uses Moodle's application-level cache (MODE_APPLICATION)
2. Cache keys are MD5 hashes of endpoint + sorted parameters
3. Cached data includes timestamp for TTL validation
4. Cache is checked before expensive DB queries
5. Fresh data is automatically cached after retrieval
6. Audit logs distinguish between cached and uncached requests

**Cache Configuration (db/caches.php):**
- Mode: APPLICATION (persistent across requests)
- Simple keys: true
- Simple data: false (stores complex arrays)
- Default TTL: 300 seconds
- Static acceleration: enabled (100 items in memory)

**Benefits:**
- Reduces database load for repeated requests
- Faster response times for cached data
- Configurable per Moodle's cache settings
- Can be purged via Moodle admin UI
- Falls back gracefully if cache unavailable
